1. 阈值从0.5降到0.3时，TP、FP、FN通常会向什么方向变化？
阈值从0.5降到0.3时，TP通常会增加，FP通常会增加，FN通常会减少。

2. 为什么在测试集上搜索最优阈值属于数据泄漏？
阈值是需要通过数据选择的超参数。在测试集上搜索阈值，等于利用测试集的真实标签调整最终决策规则，使测试集参与了模型选择。此时得到的测试 F1 已经针对该测试集优化，会过度乐观，无法继续作为无偏的泛化评估。

3. 如果0.35和0.40得到完全相同的最优 F1，当前代码会选哪一个？这个并列规则更偏向 Precision 还是 Recall？
`np.argmax()` 会返回第一个最大值的索引。当阈值按升序排列时，当前代码会选择较小的 `0.35`。较低阈值通常会预测更多正标签，减少 FN 但可能增加 FP，因此这个并列规则通常更偏向 **Recall**，而不是 Precision。

> 严格地说，`np.argmax()` 选择的是“第一个最大值”，只有在阈值数组按升序排列时，它才等价于选择较小的并列阈值。

4. 为什么每标签阈值可能提升 macro F1，却比全局阈值更容易过拟合？
全局阈值只需选择1个超参数，而每标签阈值需要选择50个超参数，自由度更高。低频标签在验证集中可能只有少量正例，某几个样本的预测变化就会显著改变该标签的最优阈值。因此，逐标签搜索容易让阈值适配验证集中的偶然波动、少数样本和标签噪声。

```text
全局阈值：1个自由度，更稳定但不够灵活
每标签阈值：50个自由度，更灵活但更容易拟合验证集噪声
```

## Day 4 新发现：单元测试的作用

本日通过 T2/T3 的填写，建立了对单元测试更准确的理解：

> 单元测试是先构造一个“正确答案已知”的小问题，再调用 tool 中的被测函数，用断言检查函数输出是否符合已知答案。

`tools/threshold_search.py` 与 `tests/test_day4_threshold.py` 的关系是：

```text
tools/threshold_search.py
    ↓
被测对象：真正执行阈值搜索

tests/test_day4_threshold.py
    ↓
检查者：构造输入、给出已知正确答案、执行断言
```

一个基础测试可以分为 Arrange–Act–Assert 三步：

```python
def test_example():
    # 1. Arrange：构造输入和答案已知的场景
    y_true = ...
    y_prob = ...

    # 2. Act：调用被测函数
    result = search_global_threshold(...)

    # 3. Assert：用预先知道的正确答案判卷
    assert result["best_threshold"] == pytest.approx(expected_threshold)
```

### 正确答案从哪里来

测试中的期望值不应该是“先运行被测函数，再把它的输出抄进 `assert`”。正确流程是：

```text
构造足够简单的输入
    ↓
通过观察或手算先得到正确答案
    ↓
把正确答案写入assert
    ↓
运行被测函数
    ↓
检查函数是否得到同样的答案
```

T2 的数据是刻意设计的：阈值0.5时，所有正样本概率都不低于0.5，所有负样本概率都低于0.5，因此在运行搜索函数之前就能确定：

```text
best_threshold = 0.5
best_f1 = 1.0
```

T3 则用于检查并列规则。三个阈值的 F1 都为1.0，而 `np.argmax()` 返回第一个最大值的索引，因此在当前排序下已知正确答案是0.3。

如果精确数值难以人工推导，还可以测试函数必须满足的性质：

```python
assert 0.0 <= result["best_threshold"] <= 1.0
assert 0.0 <= result["best_f1"] <= 1.0
assert len(result["f1_scores"]) == len(thresholds)
```

也可以将自实现结果与可信的标准库实现对比，但不能用被测函数自己的输出来证明自己正确。

### 本日工程认识

```text
测试是先构造答案已知的问题，
再用已知答案检查函数能否正确解题。
```

### 2. 三种阈值策略对比

| 阈值策略 | Micro F1 | Macro F1 |
|---|---:|---:|
| 固定0.5 | 0.8750 | 0.8667 |
| 全局最优0.45 | 1.0000 | 1.0000 |
| 每标签阈值 `[0.40, 0.45, 0.35]` | 1.0000 | 1.0000 |

固定0.5漏掉了两个真实正标签，它们的预测概率分别是0.48和0.46，都略低于0.5。全局阈值降至0.45后找回了这两个正标签，同时没有引入假阳性，因此达到完美预测。

每标签阈值也达到完美预测，但没有超过全局阈值。这说明：

> 更复杂、更灵活的阈值策略不一定带来额外收益。当一个全局阈值已经能正确分开正负样本时，增加每标签自由度只是增加了复杂性。

---

### 3. 最优阈值并不一定唯一

当前合成数据中：

```text
最大负样本概率 = 0.42
最小正样本概率 = 0.46
```

因此，任何满足下式的阈值都能实现完美分类：

$$
0.42 < t \le 0.46
$$

0.45并不是数学上唯一的最优阈值，而是当前候选网格中第一个取得最高 F1 的阈值。

每标签阈值也同理：

- 标签0：0.40是网格中第一个完美阈值。
- 标签1：0.45是网格中第一个完美阈值。
- 标签2：0.35是网格中第一个完美阈值。

因此，更准确的表述是：

> 搜索结果是“当前候选网格与并列规则下选出的最优阈值”，而不是唯一的真实最优阈值。

阈值网格的步长本身也是一个超参数：

- 步长较大：计算更快，但可能跳过更精细的最优区域。
- 步长较小：搜索更精细，但也更容易追逐验证集中的微小波动。

### 4. 第二阶段代码复盘

#### `apply_per_label_thresholds()`

```python
(y_prob >= thresholds).astype(np.int64)
```

这里利用 NumPy 广播：

```text
y_prob:      [N, K]
thresholds:     [K]
```

`thresholds[k]` 会自动与 `y_prob` 的第 $k$ 列比较，不需要手写两层循环。

#### `search_per_label_thresholds()`

外层循环遍历标签，内层循环遍历候选阈值：

```text
每个标签
    ↓
对每个候选阈值计算binary F1
    ↓
选择第一个最大F1对应的阈值
```

时间复杂度大致为：

$$
O(KTN)
$$

其中 $K$ 是标签数，$T$ 是候选阈值数，$N$ 是验证样本数。当前 $K=50$ 的规模完全可接受。

#### 无效标签回退

当验证集中某个标签全0或全1时：

```python
if np.unique(label_true).size < 2:
    continue
```

函数保留 `fallback_threshold`，不为它伪造“最优阈值”。这不是说该标签完全不能计算 F1，而是只观察到一个类别时，无法可靠判断阈值对正负区分能力的影响。

### 6. T6 带来的测试认识

T6 最初将第二个标签的预期阈值写为0.7，但工具函数返回0.5。逐阈值核对后发现：

```text
threshold=0.3 → F1=0.8
threshold=0.5 → F1=1.0
threshold=0.7 → F1=1.0
```

0.5和0.7并列最优，`np.argmax()` 会选择第一个最大值，因此正确断言是0.5。

这说明：

> 测试失败不会自动证明工具函数错误，测试输入、期望答案和边界/并列规则也需要被检查。

排查测试失败时，应依次检查：

```text
1. 被测函数实现是否错误？
2. 测试输入是否符合原设计？
3. 断言中的期望答案是否推导正确？
4. 是否存在没有考虑的边界或并列规则？
```

## Day 4 研究者视角观察

> 阈值优化带来的性能提升不一定来自模型表征能力增强，也可能只来自决策边界调整。报告结果时必须区分“模型变好了”和“后处理策略更适合验证集”，并关注阈值在独立测试集上的泛化稳定性。

> 当全局阈值已经与每标签阈值取得相同表现时，根据简约性原则应优先保留全局阈值，除非每标签策略在独立数据上证明了稳定的额外收益。
